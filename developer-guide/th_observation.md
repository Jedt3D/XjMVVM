# TH Translation Editorial Review

## Overall Assessment

งานแปลมีคุณภาพโดยรวมอยู่ในระดับใช้ได้ — โครงสร้างประโยคส่วนใหญ่ถูกต้องและเนื้อหาเชิงเทคนิคถ่ายทอดได้ครบถ้วน อย่างไรก็ตาม มีปัญหาเด่นสองชั้นที่ต้องแก้ไขก่อนเผยแพร่: (1) ความไม่สม่ำเสมอของศัพท์เฉพาะทางทั่วทั้งไฟล์ โดยเฉพาะคำว่า "การตรวจสอบอัตลักษณ์" / "การรับรองตัวตน" / "การรับรองความถูกต้อง" ที่ใช้สลับกันอย่างไม่มีรูปแบบ และ (2) รอยนิ้วมือของการแปลด้วยเครื่องยังคงปรากฏให้เห็นในหลายที่ ทั้งประโยคที่ยาวเกินไป ทับศัพท์ที่ทับซ้อนกันโดยไม่จำเป็น และสำนวนแปลกหูที่นักพัฒนาชาวไทยไม่พูดกันตามธรรมชาติ

---

## Global Issues

### G1 — ศัพท์ "Authentication" ขาดความสม่ำเสมอ (ปัญหาร้ายแรงที่สุด)

คำว่า "authentication" แปลต่างกันในแต่ละไฟล์โดยไม่มีเหตุผล:

- `09-authentication/index.md` ใช้: **"การตรวจสอบอัตลักษณ์"**
- `10-api/index.md` และ `12-protected-routes/index.md` ใช้: **"การรับรองตัวตน"**
- `12-protected-routes/index.md` ยังใช้: **"การรับรองความถูกต้อง"**
- `index.md` (หน้าหลัก) ใช้: **"ตรวจสอบสิทธิ์"**

ผู้อ่านจะสับสนว่าทั้งหมดนี้หมายถึงสิ่งเดียวกันหรือไม่ ต้องเลือกคำเดียวและใช้ทั่วทั้งเอกสาร (ดูข้อแนะนำในส่วน Glossary)

=> แก้ตามตารางคำแปลท้ายเอกสาร

### G2 — ทับศัพท์ภาษาอังกฤษทับซ้อนกันโดยไม่จำเป็น

หลายไฟล์ใช้คำทับศัพท์ในบริบทที่มีทางเลือกที่อ่านเป็นธรรมชาติกว่า เช่น "เนวิเกชัน", "ไคลเอนต์", "ตัวจัดอักษร" (แทน serializer) ปัญหานี้ไม่ได้อยู่ที่การทับศัพท์เอง (ในเอกสารเทคนิค การทับศัพท์ทำได้) แต่อยู่ที่ความไม่สม่ำเสมอ — บางครั้งใช้ไทย บางครั้งใช้ทับศัพท์สำหรับคำเดียวกัน
=> แก้ตามตารางคำแปลท้ายเอกสาร

### G3 — ประโยคยาวเกินไปในหลายไฟล์

การแปลมักรวมประโยคยาวๆ หลายประโยคของต้นฉบับภาษาอังกฤษเข้าด้วยกันโดยใช้คำเชื่อม "ซึ่ง", "ดังนั้น", "เพราะ" ซ้อนกันจนยากต่อการอ่าน นักพัฒนาไทยอ่านเอกสารเร็ว ประโยคสั้นและตรงประเด็นเป็นที่ต้องการมากกว่า
=> แก้โดยปรับภาษาให้กระชับขึ้น

### G4 — หัวข้อภาษาอังกฤษบางส่วนไม่ได้รับการแปล

หลายไฟล์ยังมีหัวเรื่องย่อยเป็นภาษาอังกฤษล้วน เช่น `## The core pattern`, `## Return types`, `## Per-request database connections` (ใน `05-database/index.md`) และส่วน `## Framework vs. ViewModels`, `## Template mirroring` (ใน `02-conventions/directory-structure.md`) ซึ่งสลับกับหัวเรื่องภาษาไทยในไฟล์เดียวกัน ดูไม่สม่ำเสมอ
=> แก้ตามตารางคำแปลท้ายเอกสาร

### G5 — "พจนานุกรม" และ "Dictionary" ใช้สลับกันในลักษณะที่อาจสับสน

บางไฟล์ใช้ "พจนานุกรม" สำหรับ Dictionary object ของ Xojo ซึ่งอาจสับสนกับพจนานุกรมภาษาแบบทั่วไป ทั้งที่ควรใช้ "Dictionary" (ทับศัพท์) ตลอดเพื่อความชัดเจน
=> แก้โดยไม่แปล ใช้ Dictionary ไปเลย รวมทั้ง Key ในบริบทของ Dictionary ด้วย

### `index.md` (หน้าบทนำ)

**ปัญหา 1** — ประโยคเปิดซับซ้อนเกินไป
> "นี่คือ **XjMVVM** (Xojo MVVM Web Framework) — เฟรมเวิร์กเว็บแอปพลิเคชันที่เรนเดอร์ฝั่งเซิร์ฟเวอร์ (SSR) สร้างบน Xojo Web 2 แทนที่จะใช้ระบบ GUI WebPage และ WebControl ที่สร้างขึ้นมาของ Xojo ทุก HTTP request จะถูกสกัดในที่ `HandleURL` และส่งไปยัง ViewModel ซึ่งจะเรนเดอร์ HTML response โดยใช้เอนจิน **JinjaX template**"

ประโยคนี้ยาวมากและมีวลี "ที่สร้างขึ้นมาของ Xojo" ซึ่งแปลกหูมาก ต้นฉบับ "built-in" ไม่ได้หมายความว่า "ที่สร้างขึ้นมา"

**คำแนะนำ:**
> "นี่คือ **XjMVVM** (Xojo MVVM Web Framework) — เฟรมเวิร์กเว็บแอปพลิเคชัน SSR ที่สร้างบน Xojo Web 2 แทนที่จะใช้ระบบ GUI WebPage และ WebControl ของ Xojo ทุก HTTP request จะถูกดักจับใน `HandleURL` และส่งต่อไปยัง ViewModel เพื่อเรนเดอร์ HTML response ด้วยเอนจิน **JinjaX template**"

**ความรุนแรง:** Moderate

### `07-routing/index.md`

ต้นฉบับ "path normalization" แต่วลีนี้แปลกมากในภาษาไทย

"Pipeline" แปลว่า "ท่อ" ได้ในบริบทอื่น แต่ในเอกสารเทคนิคโปรแกรมมิ่ง มักใช้ทับศัพท์ว่า "pipeline" หรือ "กระบวนการประมวลผล"

**ปัญหา 4** — "การเต้นรำสามขั้น" อ่านดีแต่ขาดความสม่ำเสมอกับส่วนอื่น

**คำแนะนำ:** "ทริกเกอร์ขั้นตอน redirect dance" หรือ "กระตุ้นลำดับ redirect ไปยัง XojoUnit"

**ความรุนแรง:** Major (บรรทัดที่สอง), Minor (บรรทัดแรก)

**ปัญหา 5** — "เอกลักษณ์ `request.Path`" หัวข้อที่ไม่ชัดเจน

**ความรุนแรง:** Moderate (ความถูกต้องของข้อมูล)

## Terminology Glossary Suggestions

ตารางนี้เสนอคำแปลมาตรฐานที่ควรใช้สม่ำเสมอทั่วทั้งชุดเอกสาร

| ภาษาอังกฤษ (ต้นฉบับ) | คำแนะนำสำหรับภาษาไทย | เหตุผล |
|---|---|---|
| Authentication | **การยืนยันตัวตน** | ชัดเจน ตรงความหมาย ไม่สับสนกับ "authorization" |
| Authorization | **การอนุญาตสิทธิ์** | แยกจาก authentication |
| Session | **session** (ทับศัพท์) | นักพัฒนาไทยใช้คำนี้กันอยู่แล้ว |
| Template | **เทมเพลต** | ทับศัพท์ที่คุ้นเคย ไม่ต้องใช้ "แม่แบบ" |
| Dictionary | **Dictionary** (ทับศัพท์) | ใช้เป็นชื่อเฉพาะ object ของ Xojo ไม่ใช่ "พจนานุกรม" |
| Key (of Dictionary) | **key** หรือ **คีย์** | ไม่ใช้ "กุญแจ" |
| Route | **route** หรือ **เส้นทาง** | ใช้ได้ทั้งคู่แต่ให้เลือกอย่างใดอย่างหนึ่ง |
| Redirect | **redirect** หรือ **เปลี่ยนเส้นทาง** | ทั้งคู่ใช้ได้ ควรสม่ำเสมอ |
| Request / Response | **request / response** (ทับศัพท์) | นักพัฒนาไทยคุ้นเคยกับคำเหล่านี้ |
| Middleware / Guard | **guard** (ทับศัพท์) | ในบริบทนี้ "guard" ชัดกว่า "ยาม" |
| Browser | **เบราว์เซอร์** | ใช้รูปแบบเดียวทั่วทั้งเอกสาร (มีวรรณยุกต์) |
| Server | **เซิร์ฟเวอร์** | ใช้รูปแบบเดียว |
| Pipeline | **pipeline** หรือ **กระบวนการประมวลผล** | ไม่ใช้ "ท่อ" |
| Flag (boolean) | **flag** หรือ **ตัวแปร flag** | ไม่ใช้ "ธง" |
| Inject | **inject** หรือ **เพิ่มเข้าไปอัตโนมัติ** | ขึ้นกับบริบท |
| Serialize | **แปลงเป็น** (JSON/string) | ไม่ใช้ "จัดอักษร" |
| Directory | **ไดเรกทอรี** (ไม่มีไม้จัตวา) | ใช้รูปแบบเดียว |
| Note (data entity) | **บันทึก** หรือ **Note** | สม่ำเสมอ ไม่ใช้ "หมายเหตุ" |
| Escape (HTML) | **escape** (ทับศัพท์) | ไม่ใช้ "หลีกเลี่ยง" ในบริบท HTML |
| Port (of a library) | **port** (ทับศัพท์) | ไม่ใช้ "นำเสนอ" |
| Async | **async** หรือ **แบบ asynchronous** | ไม่ทับศัพท์เป็น "อะแซงก์ก" |
| Pattern | **รูปแบบ** | "ลวดลาย" ฟังแปลกในบริบทเทคนิค |
