# ทำไมถึง XjMVVM?

หน้านี้อธิบายการตัดสินใจหลักเบื้องหลังการออกแบบเฟรมเวิร์ก การทำความเข้าใจ *เหตุผล* ทำให้ทุกอนุสัญญาในส่วนอื่น ๆ ของคู่มือนี้สมเหตุสมผล

## ทำไมต้องข้ามระบบ WebPage ของ Xojo?

Xojo Web 2 มีระบบวิดเจ็ต GUI แบบเต็ม: `WebPage`, `WebButton`, `WebTextField`, `WebListBox` และอื่น ๆ อีกมากมาย วิดเจ็ตเหล่านี้สื่อสารผ่าน WebSockets และจัดการสถานะโดยอัตโนมัติ สำหรับเครื่องมือภายในที่ง่ายซึ่งสะท้อนโมเดลวิดเจ็ต Xojo อย่างใกล้ชิด พวกมันก็ใช้ได้ดี

โครงการนี้ข้ามระบบนั้นไปอย่างสิ้นเชิง นี่คือเหตุผล:

**การควบคุม HTML/CSS แบบเต็ม** WebControls เรนเดอร์ HTML ของตัวเองด้วยชื่อคลาสและโครงสร้างของตนเอง คุณไม่สามารถใช้ระบบการออกแบบแบบกำหนดเองหรือรวมเฟรมเวิร์ก CSS เช่น Tailwind ได้อย่างง่ายโดยไม่ต้องต่อสู้กับผลลัพธ์ของการควบคุม ด้วยเทมเพลต JinjaX คุณเขียน HTML ด้วยตัวเอง — แท็กทุกแท็ก คลาสทุกคลาส แอตทริบิวต์ข้อมูลทั้งหมดเป็นของคุณ

**ความหมาย HTTP มาตรฐาน** หน้า WebControl เป็นเซสชัน WebSocket ที่มีสถานะ พวกมันไม่ทำตัวเหมือนแอป HTTP มาตรฐาน: การจดหมายระหว่างประเทศไม่น่าเชื่อถือ ประวัติเบราว์เซอร์ยุ่งยาก API REST ยากขึ้น แนวทาง SSR หมายความว่าทุก URL เป็นเอนด์พอยต์ HTTP ที่เหมาะสม — คาดเดาได้ แคชได้ เชื่อมโยงได้

**เทมเพลตเป็นมิตรกับนักออกแบบ** เทมเพลต HTML อาศัยอยู่ในโฟลเดอร์ `templates/` นักออกแบบหรือนักพัฒนาฝั่งหน้าสามารถแก้ไขได้โดยใช้โปรแกรมแก้ไขข้อความใด ๆ ที่มีการสนับสนุน IDE แบบเต็ม (การไฮไลต์ไวยากรณ์ Emmet Prettier) พวกเขาไม่เคยสัมผัสโค้ด Xojo

**ไม่มีสถานะเซสชันในเลเยอร์ UI** WebControls ของ Xojo มีสถานะเซสชันโดยนัย ViewModels ของเราไม่มีสถานะต่อคำขอ — พวกเขาได้รับคำขอ ทำงาน และเขียนการตอบสนอง สถานะถาวรเพียงอย่างเดียวอยู่ใน `Session` (คลาส `WebSession`) และฐานข้อมูล

## ทำไมต้อง MVVM มากกว่า MVC?

MVC แบบคลาสสิก (Model-View-Controller) มี Controller ที่รับผิดชอบทั้งตรรกะการจัดเส้นทางและการจัดวาง Model ใน Xojo หมายความว่าคลาส Controller ขนาดใหญ่หนึ่งคลาสจัดการทุกเส้นทาง

XjMVVM ใช้ MVVM เพื่อแยกข้อกังวลให้สะอาดขึ้นสำหรับสถาปัตยกรรมต่อเส้นทาง:

| เลเยอร์ | คลาส | ความรับผิดชอบ |
|---|---|---|
| **View** | ไฟล์ `.html` Jinja2 | การนำเสนอเท่านั้น — ไม่มีตรรกะทางธุรกิจ |
| **ViewModel** | คลาสหนึ่งต่อเส้นทาง | ประสานงาน Model สร้างบริบทเทมเพลต |
| **Model** | คลาสการเข้าถึงข้อมูล | การค้นหาฐานข้อมูล คืนค่าอ็อบเจ็กต์ `Dictionary` |

ViewModel หนึ่งต่อเส้นทาง (เช่น `NotesListVM`, `NotesCreateVM`) ให้แต่ละคลาสมีขนาดเล็กและมุ่งเน้น การเพิ่มฟีเจอร์ใหม่หมายถึงการเพิ่มไฟล์ใหม่ ไม่ใช่การปรับเปลี่ยนไฟล์ที่มีอยู่

ViewModels ไม่เคยอ้างอิงชื่อเทมเพลตในตรรกะทางธุรกิจ Model ไม่เคยอ้างอิง ViewModels สิ่งนี้ทำให้แต่ละเลเยอร์สามารถทดสอบและแทนที่ได้อย่างอิสระ

## ทำไมต้อง JinjaX?

JinjaX เป็นพอร์ตของเอนจิน Jinja2 ของ Python ที่เขียนด้วย Xojo แบบบริสุทธิ์ มันถูกเลือกเพราะ:

**Jinja2 เป็นมาตรฐานในทางปฏิบัติ** นักพัฒนาที่คุ้นเคยกับ Flask Django Ansible หรือ Saltstack รู้ไวยากรณ์อยู่แล้ว เส้นโค้งการเรียนรู้น้อยมาก

**มันคือ Xojo บริสุทธิ์** ต้นไม้ต้นฉบับของ JinjaX เต็มที่อยู่ภายใต้ `JinjaXLib/` ไม่มีการพึ่งพาไลบรารี `.xojo_library` ไบนารี ซึ่งหมายความว่ามันใช้งานได้กับรูปแบบโครงการข้อความของ Xojo (ไฟล์ `.xojo_code`) และสามารถตรวจสอบและดีบัตได้อย่างเต็มที่

**มันปลอดภัยเธรดหลังจากการตั้งค่า** `JinjaEnvironment` ถูกเริ่มต้นเพียงครั้งเดียวใน `App.Opening()` แล้วจึงอ่านเท่านั้น แต่ละคำขอสร้าง `CompiledTemplate` และ `JinjaContext` ใหม่ — อ็อบเจ็กต์เหล่านี้ต่อคำขอ ไม่เคยใช้ร่วมกัน

**การสืบทอดเทมเพลตใช้งาน** `{% extends %}`, `{% block %}` และ `{% include %}` ให้คุณสร้างเลเยต์ฐานครั้งเดียวและสร้างหน้าทั้งหมดจากนั้น

## สัญญาข้อมูล Dictionary

นี่คือกฎที่สำคัญที่สุดในระบบทั้งหมด

JinjaX แก้ไขสัญกรณ์จุด (`{{ note.title }}`) โดยเรียก `EvaluateGetAttr` บนอ็อบเจ็กต์ ใน Xojo `EvaluateGetAttr` ถูกใช้งานเฉพาะสำหรับ `Dictionary` — จะค้นหากุญแจตามชื่อ อ็อบเจ็กต์คลาส Xojo แบบกำหนดเองไม่รองรับ `EvaluateGetAttr` ดังนั้นการเข้าถึงจุดจึงเรนเดอร์เป็นค่าว่างเงียบ ๆ

**กฎ:** ข้อมูลทั้งหมดที่ส่งไปยังเทมเพลตต้องเป็น `Dictionary` หรืออาร์เรย์ `Variant()` ของอ็อบเจ็กต์ `Dictionary` Model ส่งคืนหนึ่งในสองประเภทนี้เสมอ — ไม่มีอินสแตนซ์คลาสแบบกำหนดเอง

## Post/Redirect/Get

การส่งแบบฟอร์มทั้งหมดปฏิบัติตามรูปแบบ **Post/Redirect/Get (PRG)**

## การแยกเซสชัน

`App` เป็นซิงเกิลตันที่ใช้ร่วมกัน — คำขอของผู้ใช้ทั้งหมดทำงานในอ็อบเจ็กต์ `App` เดียวกัน อย่าเก็บข้อมูลเฉพาะผู้ใช้บนคุณสมบัติ `App` สถานะต่อผู้ใช้ไปในคลาส `Session`
